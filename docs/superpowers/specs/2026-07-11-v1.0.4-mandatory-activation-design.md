# v1.0.4 Mandatory Activation Recovery Design

## Status and authority

This design implements GitHub issue #532 and milestone #16. The owner directive requires mandatory API-backed activation before useful review or provider work, a verified immutable replacement release, and replacement-first cleanup of predecessor distribution surfaces.

The supported v1.0.4 release may claim enforcement only for the official npm CLI and the future official desktop distribution. It cannot recall or technically prevent use of public source, forks, clones, caches, edited installed JavaScript, or already-downloaded artifacts.

## Current failure

v1.0.3 treats licensing as optional policy:

- `license.enabled=false` bypasses the review gate.
- `license.publicReposFree=true` bypasses activation for public repositories.
- any HTTPS or loopback license API can be configured.
- an unsigned, user-writable cache can authorize work during an outage and can be accepted without a stored key.
- provider verification is reachable from CLI and dashboard without license admission.
- Desktop stores a key under a different Keychain identity, reports service activation as pending, and permits public-mode onboarding without activation.

Changing defaults alone is insufficient because legacy/user config can restore every weakening value.

## Approaches considered

### 1. Locked online policy and central admission guard — selected

The official distribution owns a non-user-editable effective policy: canonical production API, mandatory activation for every repository visibility, and zero offline cache authority. Legacy config continues to load for upgrade safety, but weakening fields are ignored and surfaced as migration diagnostics. Every useful-work seam consumes one shared admission result.

This is the smallest design that removes the current bypasses without stranding users before activation.

### 2. Remove legacy fields and reject old config

Removing `enabled`, `publicReposFree`, and configurable API policy is easy to reason about, but a v1.0.3 config would fail during load. That would also block `license activate`, creating a recovery loop that requires manual config surgery. This approach is rejected for v1.0.4.

### 3. Server-signed offline entitlement

A signed, machine-bound entitlement could support bounded offline work without trusting a writable cache. It requires signing-key custody, verification-key pinning, replay/clock/rotation design, API changes, and broader security proof. This is deferred until after the emergency replacement. A local HMAC is not acceptable because the user possesses the license key.

## Effective production policy

Create a focused production-policy module with these immutable values:

- activation is mandatory;
- license API is `https://neondiff-license.fly.dev`;
- public, private, internal, and unknown repository work all require active entitlement;
- live validation is required before each useful-work admission;
- offline grace is zero;
- an entitlement cache is diagnostic only and never grants useful work;
- user config may select approved secret storage and non-sensitive filesystem locations, but may not weaken policy or redirect the API.

Tests inject a policy and API dependency through typed function arguments. No environment variable, hidden CLI flag, alternate config key, or release-channel switch may weaken the official runtime.

Legacy v1.0.0–v1.0.3 config remains parseable. Effective config replaces weakening values and reports a redacted migration warning. Activation must work without requiring the user to add an API URL manually.

## Activation and admission data flow

1. Setup accepts the license secret through bounded stdin or native secure input, never argv, logs, issue text, or evidence.
2. The client calls the canonical `/v1/license/activate` endpoint.
3. A successful, schema-valid response is persisted only after approved secret storage succeeds.
4. Before useful work, the client reads the stored secret and performs live `/v1/license/validate` against the canonical API.
5. Only an active, scope-valid response permits exactly the requested operation.
6. Missing key, wrong scope, expired/revoked/invalid status, malformed response, timeout, DNS/network failure, 429, 5xx, clock error, fake API configuration, forged cache, or fingerprint mismatch blocks before provider input, checkout, worktree creation, model calls, or GitHub mutation.
7. Cache metadata may be shown as redacted diagnostics but cannot change admission.

## Shared useful-work admission

Add a single `requireActiveProductionLicense` contract that returns a typed redacted decision and is used at these seams:

- review worker before file listing, checkout, worktree creation, provider calls, or posting;
- `review-pr`, `run-once`, retry paths, and live daemon cycles through the worker seam;
- daemon start before a useful runtime can launch;
- `providers verify` before stdin is read and before provider selection/network;
- `providers doctor --smoke true` before a provider request;
- dashboard `/api/provider/verify` before request-body reading;
- desktop provider verification, daemon start, review actions, and update checks.

Unauthenticated setup-safe surfaces are explicitly allowlisted:

- help and version;
- config initialization, inspection, migration diagnostics, and non-secret setup patches;
- license activate, status, refresh, and deactivate;
- dashboard setup/status shell and activation route;
- metadata-only provider list/doctor;
- daemon stop/status;
- deterministic offline fixture/evaluation commands that cannot invoke providers, inspect customer repositories, or post output.

Everything else defaults to blocked until live activation.

## Cache and offline behavior

v1.0.4 is online-only for useful work. A fresh legitimate cache does not grant an outage exception. The client must:

- reject cache-only authority when no key is stored;
- reject cache/key fingerprint mismatch;
- reject forged, malformed, expired, revoked, or stale cache state;
- preserve redacted cache metadata only for status diagnostics;
- return a clear `offline` or service classification while keeping the gate closed.

Setup, status, logs, and remediation guidance remain available offline.

## CLI, dashboard, and desktop onboarding

### CLI

Activation accepts bounded secret stdin. JSON output contains only status, timestamps, scope, plan, and a short fingerprint. Provider verification denial happens before provider-key stdin is read.

### Dashboard

Add a same-origin activation route using a bounded JSON body and the shared activation client. The response is redacted. License state is `blocked`, never `degraded`, until live validation succeeds. Provider verification and first-review controls remain disabled until then.

### Desktop

Replace local-only pseudo-activation with a real bridge to the installed CLI activation contract. The desktop sends secret bytes over stdin, validates a strict redacted response, and stores the secret in Keychain only after API success. If Keychain persistence fails, it performs compensating deactivation with the transient secret.

Align CLI and Desktop on Keychain service `com.electricsheephq.NeonDiffDesktop.secrets` and account `license/default`, with deterministic redacted migration/probe behavior for the legacy CLI identity.

Desktop entitlement state becomes typed: `missing`, `activating`, `active`, `expired`, `revoked`, `offline`, or `failed`. Merely storing a key is never active. Onboarding order is activation, provider, repository, daemon/review. Remove the public-mode bypass and disable provider/daemon/review/update actions whenever state is not active.

## Upgrade behavior

A real v1.0.3 config must:

- load under v1.0.4 without manual editing;
- report that optional/public-free/fake-API/offline-grace fields were superseded;
- expose activation and setup remediation immediately;
- block useful work until live activation;
- keep existing daemon jobs fail-closed rather than running under legacy flags.

The release must include isolated upgrade smoke from installed v1.0.3 to the candidate v1.0.4 package.

## Test and evaluation design

Red-first tests cover:

- empty and legacy config resolving to the locked policy;
- attempts to disable enforcement, restore public-free mode, weaken private checks, change the API, or enable grace;
- forged cache with no key, wrong key, fingerprint mismatch, legitimate cache during outage, and malformed cache;
- active, expired, revoked, invalid, scope mismatch, clock error, timeout, 429, 5xx, and malformed API responses;
- public/private/unknown visibility in dry and live review paths;
- zero file-list, checkout, worktree, provider, or GitHub-post calls on denial;
- provider CLI denial before secret stdin or network;
- dashboard denial before body read and provider call;
- desktop onboarding, stale async completion, Keychain persistence failure, legacy identity migration, and action enablement;
- real v1.0.3 config upgrade behavior;
- release gate rejection without exact installed-package activation/validate/deactivate/no-bypass evidence.

Each successful admission test proves exactly one intended operation is allowed after live validation.

## Release and cleanup gates

No release is published until all of the following bind to one exact head:

- focused and full tests, build, scans, and action lint;
- independent security and product-spec review;
- production API health plus a real redacted disposable-entitlement activate/validate/deactivate or revoke flow;
- isolated clean install and v1.0.3 upgrade smoke;
- CLI, dashboard, desktop, public/private/unknown, offline, forged-cache, and fake-API no-bypass evidence;
- real non-prerelease GitHub Release and immutable npm package;
- workflow-owned `latest` promotion and bounded registry-convergence verification.

Only after replacement proof may cleanup start. npm policy currently prevents ordinary unpublish because weekly downloads exceed its threshold, so v1.0.0–v1.0.3 must be version-specifically deprecated unless npm Support grants an exception. GitHub predecessor Releases/tags and the 16 non-expired Actions artifacts may then be deleted against a private exact-ID ledger.

The final audit must report remaining npm versions and deprecation notices, dist-tags, GitHub Releases/tags/artifacts, installed behavior, and the unavoidable public-source/history boundary.

## Non-goals

- preventing edited public source or forks from running;
- signed/notarized Mac distribution or Sparkle/appcast maturity;
- browser/native parity or v1.1 completion;
- restoring offline useful work without server-signed entitlements;
- deleting any predecessor before v1.0.4 replacement and `latest` convergence proof.
