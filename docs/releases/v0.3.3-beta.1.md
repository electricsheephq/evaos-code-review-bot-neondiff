## Purpose

Promote the first-class operator CLI for the live beta worker and harden the
diagnostics found by current-head automated review before merge. This patch
improves the operator's ability to answer "why did or did not this PR get
reviewed?" without mutating runtime state or broadening live review scope.

## Changes

- Add `status`, `agents`, `queue`, `coverage`, `cooldowns`, and `why` operator
  commands.
- Add `docs/operator-cli.md` with read-only usage guidance.
- Add post-build executable-bit handling for the CLI entrypoint.
- Harden `why` so provider-deferred and stale-head states are not hidden behind
  generic processed rows.
- Scope repo provider cooldown inventory when `cooldowns --repo` is used.
- Add stale-head status gating so aggregate operator health cannot stay green
  when coverage found stale heads.
- Preserve lease diagnostics for old state databases that predate `owner_pid`.
- Treat `EPERM` PID probes as live process ownership and prefer dead-owner
  lease diagnostics over generic expiry when both apply.

## Runtime Caveats

- This release changes operator diagnostics and release-health interpretation.
  It does not expand monitored repos, App permissions, review concurrency,
  native ZCode skill mode, MCP access, command-trigger policy, or posting
  policy.
- The operator CLI is read-only except for existing explicit retry/retire
  commands; this patch does not add automatic cleanup or mutation.
- Queue states remain coverage-derived until #83 adds the provider-aware
  bounded scheduler.
- Post-release gates must run `release:status`, `coverage-audit`, and
  `provider-cooldowns --expired-only true` before the release is called green.
