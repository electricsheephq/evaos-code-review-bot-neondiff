## Purpose

Promote the review-status repair for PR heads that are already durably posted in
SQLite but whose public sticky status comment can remain non-terminal.

This release targets the agent-blocking failure mode from `#436`: downstream
agents wait on a sticky `queued`/`in_progress` review-status comment even though
`processed_reviews.status=posted` and the durable queue/readiness surfaces have
settled.

## Included Changes

- Add parser helpers for existing review-status marker state so duplicate
  scheduler cycles can detect and repair only non-terminal bot-owned markers.
- Repair scheduler duplicate processed-head cycles when processed review truth
  is posted and the existing sticky marker is `queued`, `in_progress`, or
  `provider_deferred`.
- Repair direct/manual review reconciliation when no active queue row remains
  but processed review truth is already posted.
- Restore missing or non-terminal readiness rows from processed truth during
  direct settled-head repair, while preserving already-settled readiness.
- Preserve no-spam behavior for already-terminal sticky markers.

## Caveats

- This release does not publish a new npm package; the public npm package
  remains `neondiff@0.4.30-beta.1`.
- This release does not change provider policy, model prompts, review quality
  calibration, issue-enrichment allowlists, monitored repos, or GitHub App
  permissions.
- This release does not cut stable `v1.0.0`, claim GA, or claim human-review
  replacement.
- The public sticky status comment is repaired only when existing durable
  processed truth proves the head was already posted.
