## Purpose

Promote the provider-aware bounded review scheduler foundation for the live beta
worker. This patch gives the daemon a durable review queue with provider, org,
repo, and manual-command capacity controls so one stalled or rate-limited review
does not hide eligible work in other monitored repos.

## Changes

- Add the provider-aware scheduler path and durable `review_queue_jobs` state.
- Add per-provider/org/repo queue leasing with lease expiry and stale lease
  requeue.
- Preserve provider cooldown windows on thrown ZCode/provider errors and retry
  provider-deferred jobs with the failed-head retry policy.
- Keep dry-run queue jobs non-terminal so dry-run evidence cannot masquerade as
  posted review evidence.
- Keep the legacy `queue` command output shape while adding `durableQueue`.
- Add release-status/operator visibility for durable queue failures,
  retryable provider-deferred jobs, and active daemon cycles.
- Gate the legacy provider-cooldown retry lane off when the scheduler owns
  queued retries.

## Runtime Caveats

- This release changes queueing, retry, release-health interpretation, and
  scheduler-owned cooldown handling.
- It does not expand monitored repos, GitHub App permissions, native ZCode
  skill mode, MCP access, repo memory, GitNexus context, auto-merge, approvals,
  or posting policy.
- The scheduler remains bounded by the live config. Provider capacity is the
  global API throttle; org/repo caps decide which jobs are allowed to spend that
  capacity.
- Provider cooldown rows still mean provider degradation was contained, not that
  the affected PR head was reviewed. Expired cooldown rows remain actionable
  backlog.
