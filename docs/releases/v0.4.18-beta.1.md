## Purpose

Promote the ZCode hard-timeout retry visibility fix for live beta operation,
and include the already-merged issue-enrichment readiness fix now present on
`main`. The timeout lane is aimed at preventing ZCode `spawnSync` timeout
failures from becoming anonymous failed rows or invisible release blockers.
The enrichment lane keeps issue scans issue-only and improves preflight checks
without enabling live issue comments or expanding active allowlists.

## Included Changes

- Adds structured ZCode hard-timeout failure errors with `retry_attempt`,
  `timeout_ms`, retryable/exhausted classification, and redacted original error
  context.
- Preserves ZCode subprocess `code`, `signal`, and `status` on wrapped failures
  so Node timeout shapes are classified even when message text varies.
- Surfaces ZCode timeout failed queue counts in release-status, operator
  status, and runtime inventory gates.
- Emits exact `retry-failed` commands only for retryable timeout jobs with
  durable queue coordinates.
- Falls back to inspect-only `queue --state failed` guidance for exhausted
  timeout jobs.
- Anchors timeout parsing to the structured prefix so retired failed-head
  `previous_error` payloads do not inflate retry attempts.
- Adds issue-enrichment GitHub Issues API read checks to `doctor` and
  `doctor github`.
- Scans only open issue records for enrichment and filters PR-shaped `/issues`
  records while paging through bounded issue windows.
- Preserves watermark safety by treating scans stopped after enough issue
  records as truncated.
- Keeps provider cooldown, scheduler, GitNexus, PR review allowlists, issue
  enrichment allowlists, and live posting behavior unchanged.
