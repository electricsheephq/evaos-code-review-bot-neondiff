## Purpose

Promote the Sprint B read-only operator dashboard for live beta operations. The
dashboard gives maintainers one bounded command to inspect coverage-audit rows,
durable queue rows, review-readiness lifecycle rows, PR links, evidence paths,
and blocked/active/healthy status without hand-querying SQLite or launchd logs.

This release closes `#27`. It does not expand monitored repositories, GitHub
App permissions, native ZCode skills, MCP, memory packets, GitNexus context,
auto-merge, approvals, repo mutation, or live concurrency.

## Changes

- Add `dashboard` to the operator CLI with JSON and human output.
- Merge coverage audit, durable queue, and review-readiness rows into a single
  per-head view with stable status precedence.
- Keep healthy queued/running/reviewing work visible without failing the
  dashboard health gate.
- Preserve failed, stale, provider-deferred, proof-blocked, check-blocked,
  needs-fix, and awaiting-re-review rows as blocking operator states.
- Attach evidence-path hints for reviewed PR heads and command-triggered work.
- Support filters for `--repo`, `--status`, `--priority`,
  `--stale-head-reason`, `--limit`, and `--job-limit`.
- Redact secret-like strings from dashboard JSON and human output.
- Document the operator dashboard in `docs/operator-cli.md`.

## Runtime Notes

- Release promotion should restart or kick the launchd job after the release
  packet commit is tagged so `release:status` proves launchd is running the
  tagged head.
- `dashboard --status needs_fix` is expected to exit nonzero while real PRs
  still need author fixes, such as already-posted `REQUEST_CHANGES` reviews.
- A dashboard row can be active without blocking the release gate. Failed,
  stale, provider-deferred, proof-blocked, check-blocked, needs-fix, and
  awaiting-re-review rows remain blocking.
- The first post-promotion health check should confirm zero uncovered eligible
  heads, zero expired provider cooldowns, no failed queue jobs, a fresh
  heartbeat, and a dashboard smoke result.
