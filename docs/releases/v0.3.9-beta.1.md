## Purpose

Promote sticky review-status comments for the live beta reviewer. The release
adds a marker-backed, App-authored PR status comment that is created when a PR
head enters the durable review queue and updated as the queue job moves through
`queued`, `in_progress`, terminal, failed, stale, or provider-deferred states.

The goal is operator and agent visibility: humans can tell whether evaOS review
is waiting, running, deferred, completed, skipped, failed, or obsolete for the
specific PR head without relying on stale recap comments.

This release does not expand monitored repos, GitHub App permissions, ZCode
tools, native skills, MCP, memory packets, GitNexus context, auto-merge,
approvals, repo mutation, or live `reviewConcurrency.maxActiveRuns`.

## Changes

- Add stable hidden markers for sticky review-status comments:
  - identity marker: repo, PR number, and head SHA
  - state marker: current status and update timestamp
- Add `reviewStatusComment` config controls, defaulting safely for existing
  runtime paths.
- Update scheduler lifecycle handling to post or update status comments for:
  - queue entry
  - in-progress review
  - completed review
  - provider-deferred review
  - stale-head retirement
  - closed or merged before review
  - skipped and failed terminal states
- Keep public status comments App-authored and head-specific.
- Preserve durable queue state before terminal status-comment side effects for
  stale and closed jobs.
- Map duplicate processed-head status from stored `processed_reviews` outcome
  instead of assuming every processed head completed successfully.
- Deduplicate active `:after-terminal:` retry jobs for repeated retry enqueue
  attempts.
- Settle attached RepoSticky session jobs during scan-time stale and closed
  queue retirement.
- Keep sticky markers outside whole-body secret redaction while still redacting
  user-controlled public fields.
- Add focused regression coverage for failed review reruns so an already failed
  terminal status is not resurrected to `in_progress`.
- Document the status-comment lifecycle in `docs/review-status-comments.md`.

## Runtime Caveats

- Sticky status comments are an operator visibility surface, not a merge gate by
  themselves.
- A terminal status comment is scoped to one PR head. New heads must receive new
  queue/review state.
- `provider_deferred` means evaOS intends to retry according to cooldown policy;
  it is not a completed review.
- `failed` requires retry or operator attention. It must not be interpreted as
  a successful review.
