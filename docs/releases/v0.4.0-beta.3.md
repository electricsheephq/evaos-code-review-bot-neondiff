## Purpose

Fix a live release-status regression found during `v0.4.0-beta.2` promotion:
scheduled provider cooldowns were stamped from scheduler cycle start time. When
ZCode/provider attempts took longer than the short overload cooldown window,
the queue could record a `provider_deferred` row whose `next_eligible_at` was
already expired as soon as the failure was handled.

## Changes

- Add an injectable scheduler event clock.
- Use failure-handling time for scheduled provider cooldown records, queue
  `provider_deferred` transitions, readiness transitions, and status-comment
  timestamps.
- Preserve deterministic scheduler tests by defaulting the event clock to the
  provided test `now` when tests inject it, and to fresh wall-clock time in
  runtime.
- Add a regression test for a slow scheduled provider failure where cycle start
  is `T0`, failure handling occurs at `T0+3m`, and cooldown expiry is recorded
  after `T0+3m`.

## Runtime Notes

- This release does not eliminate Z.ai provider overloads. It ensures slow
  overload failures renew provider cooldowns from the time the failure is
  observed, so release/status gates do not immediately see a stale cooldown
  timestamp.
- Post-promotion must restart launchd and prove `release:status` against the
  release tag target SHA.
- If the provider continues returning overload after promotion, status may show
  active provider cooldowns. That is contained provider degradation, not user
  quota exhaustion and not proof the affected PR heads were reviewed.
