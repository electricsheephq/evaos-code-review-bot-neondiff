## Purpose

Ship the issue-enrichment rollout safety layer before any live issue-comment
enablement. Issue enrichment now has an independent activation watermark per
repo, keeps issue-enrichment allowlisting separate from PR monitoring, and
prevents newly allowlisted repos from sweeping historical open issue backlogs by
default.

## Changes

- Add `issue_enrichment_repo_watermarks` SQLite state keyed by repo.
- Baseline newly allowlisted issue-enrichment repos on first activation when
  `processExistingOpenIssuesOnActivation=false`.
- Keep explicit historical backfill as a separate config path through
  `processExistingOpenIssuesOnActivation=true`.
- Use repo watermarks as `since` bounds on later scans so issue enrichment
  handles new or changed issues rather than all existing open issues.
- Hold watermarks on deferred, failed, or page-truncated scans so unprocessed
  issue work is not stranded behind an advanced cursor.
- Keep dry-run scans read-only against issue-enrichment watermarks.
- Surface `baselinedRepos` and `truncatedRepos` in issue-enrichment summaries.
- Document the activation-watermark behavior in the operator CLI guide.
- Preserve active live defaults: issue enrichment disabled, empty issue
  allowlist, and `postIssueComment=false`.

## Runtime Notes

- This release does not enable live issue enrichment. It only makes the future
  activation path bounded and repo-throttled.
- Issue-enrichment allowlisting is intentionally separate from PR monitoring.
- New issue-enrichment repos default to a watermark baseline instead of a
  historical open-issue sweep.
- Per-repo throttle and concurrency config remains available independently from
  PR review scheduling.
- Live issue comments remain disabled until a later explicit allowlist,
  permission, and throttle promotion.
- This release does not expand GitHub App permissions.
- This release does not enable native ZCode skills, MCP, memory, tools,
  auto-merge, approvals, target-repo mutation, labels, reviewers, assignments,
  or live issue comments.
- This release does not increase `reviewConcurrency.maxActiveRuns`; live remains
  one in-flight ZCode review by default.
