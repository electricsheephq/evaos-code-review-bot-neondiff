## Purpose

Add durable, operator-visible review readiness state for each reviewed PR head so
future dashboards and status surfaces can distinguish queued, reviewing,
provider-deferred, needs-fix, ready-for-human, stale, closed, skipped, failed,
and manual-command states without scraping sticky comments.

This release closes the core `#21` lifecycle-state slice. It does not enable a
dashboard UI by itself and does not widen runtime policy: live concurrency,
GitHub App permissions, monitored repos, ZCode read-only behavior, and posting
policy remain unchanged.

## Changes

- Add a `review_readiness` SQLite table keyed by `{repo, pull_number, head_sha}`
  with state, reason, timestamps, review URL/event, and manual command metadata.
- Record readiness transitions from scheduler enqueue, lease, terminal review,
  stale-head, closed-PR, provider-cooldown, failed-review, and manual-command
  paths.
- Preserve duplicate-head suppression while backfilling readiness from existing
  processed review truth and active queue jobs.
- Mark older superseded head rows stale without mutating already-terminal stale
  or closed rows.
- Preserve provider-deferred readiness during cooldown retry windows.
- Keep manual `review` and `re-review` jobs visible as reviewing when work
  starts, while `stop` and `explain` remain non-review commands.
- Align impossible leased `skipped_policy` results with queue-job failure state
  so readiness and queue truth do not diverge.
- Document the new readiness state machine in `docs/review-status-comments.md`.
