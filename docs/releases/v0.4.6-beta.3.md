## Purpose

Fix a live runtime loop where repeated provider overload responses could keep
the daemon red during peak windows by making the same PR heads eligible again
too quickly and by continuing to spend provider work in the same scheduler
cycle after a provider cooldown was already known.

## Changes

- Add bounded exponential cooldown backoff for repeated `provider_overloaded`
  / `1305` failures on the same PR head.
- Persist provider overload retry metadata in skipped processed rows:
  `retry_attempt`, provider code, and retry-after milliseconds when available.
- Stop the current scheduler provider batch after the first provider cooldown,
  then defer unstarted leased jobs with the same cooldown timestamp instead of
  immediately returning them to `queued`.
- Prefer the durable processed/repo cooldown when setting provider-deferred
  queue `nextEligibleAt`, so queue state matches recorded evidence.
- Escalate automatic `electricsheephq/evaos-code-review-bot` PR jobs to
  priority `1`, while preserving existing stale-head, base-change, and closed
  PR guards before any review work can run.
- Add focused scheduler, worker, state, and config tests for provider overload,
  cooldown metadata, self-repo priority, and closed self-repo deferred PRs.
