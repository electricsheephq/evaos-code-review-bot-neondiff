## Purpose

Add a measure-only provider-throttle telemetry report so operator incidents can
separate provider request rate limits, overloads, quota exhaustion, GitHub or
network dependency failures, and unknown provider errors without treating all
provider failures as the same runtime state.

## Included Changes

- PR `#188`: adds `provider-throttle-report` with repo, provider, hourly,
  provider-code, retry-outcome, peak-window, and known-limitation sections.
- Preserves provider-deferred retry metadata when a queued job later posts,
  including fallback provider-code recovery from `processed_reviews`.
- Hardens report parsing for SQLite UTC text, offsetless ISO timestamps, legacy
  queue schemas, provider-id sanitization, generic network matching, opaque
  provider-id token shapes, and unrelated local overload/fetch messages.
- Keeps the report advisory only: `recommendedPolicy` remains `measure_only`.

## Caveats

- `100yenadmin/Lossless-Codex-Orchestrator-LCO#423` was actively moving during
  the release gate. Earlier heads were reviewed or retired stale; the current
  head `c13a1e2156a0f2813c99a9ecb402b715839442ef` was explicitly backfilled
  before final coverage passed.
- The report is current-state telemetry, not an automatic throttle policy.
  Mixed-cause errors are assigned one primary summary category while extracted
  provider codes remain visible.
- Queue events use durable queue timestamps, so retry/update time can differ
  from first-observed throttle time until a future event-history table exists.
