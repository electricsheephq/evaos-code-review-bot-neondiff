## Purpose

Fix a live `release:status` edge case discovered during `v0.4.0-beta.3`
operation: an expired per-head provider cooldown row could remain red even when
the same `{repo, pull_number, head_sha}` already had an active queue retry.

## Changes

- Select `pull_number` and `head_sha` when reading provider cooldown rows from
  `processed_reviews`.
- Count an expired provider cooldown row as covered only when a matching
  `review_queue_jobs` row is `leased` or `running` for the exact same current
  head.
- Preserve active global provider-cooldown coverage behavior.
- Keep malformed cooldown timestamps, terminal rows, queued rows,
  provider-deferred rows, unmatched heads, expired leases, and stale null leases
  actionable instead of hidden.
- Match runtime lease expiry behavior by using configured
  `reviewConcurrency.leaseTtlMs` for legacy/null `lease_expires_at` rows.

## Runtime Notes

- This release does not eliminate Z.ai provider overloads. It makes operator
  status distinguish actionable expired cooldowns from cooldowns already covered
  by active retry work.
- `release:status` should still fail when provider-deferred queue work is
  retryable and not yet covered by a running/leased job.
- Post-promotion must restart launchd and prove `release:status` against the
  release tag target SHA.
