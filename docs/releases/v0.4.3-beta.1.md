## Purpose

Close the opt-in finishing-touch command lane without enabling autonomous target
repo mutation. Maintainers can ask for draft proposals such as risk explanations,
test suggestions, docs/docstrings, changelog notes, or review-ready checklists,
but the bot records draft-only output and does not call ZCode, fetch PR files,
push branches, commit, approve, merge, assign reviewers, or apply labels.

## Changes

- Add finishing-touch command parsing for explicit bot-mentioned commands:
  `generate tests`, `generate docs`, `generate docstrings`,
  `simplify suggestion`, `changelog draft`, `explain risk`, and
  `make review-ready`.
- Require a matching repo-profile `finishingTouches` opt-in before a draft
  command is accepted.
- Persist `finishing_touch_drafts` rows keyed by repo, PR, head SHA, and command
  comment id.
- Add `finishing-touch-dry-run` CLI support for local/operator proofs.
- Keep finishing-touch commands out of ZCode/provider review leases by recording
  terminal `command_recorded` queue rows and processing the draft path directly.
- Bind manual draft handling to the queued command comment id so newer comments
  cannot drift the leased/manual command target.
- Fail closed on untrusted authors, stale head/base, dirty deterministic
  worktrees, secret-like draft output, and ambiguous/bare command aliases.
- Keep `stop` precedence over later draft-only comments until a later trusted
  `review` or `re-review` command intentionally resumes review.
- Redact finishing-touch CLI JSON output.
