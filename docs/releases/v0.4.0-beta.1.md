## Purpose

Add bounded durable per-repo memory packets so review runs can receive
repository-specific notes, false-positive fingerprints, and recent packet
provenance without enabling native ZCode skills, MCP, memory, tools, shell, or
target-repo mutation.

This release closes the core `#81` repo-memory slice. It also hardens scheduler
handling for already-processed skipped heads so provider-cooldown skips and
historical skipped rows do not become false failed durable queue jobs.

## Changes

- Add `repoMemory` config with disabled-by-default packet controls, byte/note
  limits, note TTL validation, false-positive fingerprint limits, and
  deterministic packet hashing.
- Add `src/repo-memory.ts` and SQLite repo-memory state for read-only packet
  builds, note selection, packet provenance, and redaction summaries.
- Add `build-memory-packet` to the operator CLI, including dry-run/default
  read-only behavior and explicit `--record-build true` provenance recording.
- Inject repo-memory packets into ZCode review prompts only when enabled and
  safely built; secret-containing packets fail closed.
- Preserve false-positive suppression fingerprints across packet budget
  overflow and require canonical `finding:<sha256>` fingerprints with bounded
  future expiries.
- Harden packet redaction for token-like URLs, AWS keys, Slack tokens, cookie
  headers, private-key bodies, and raw packet output.
- Add deterministic scheduler reconciliation for historical failed queue rows
  whose only error is `processed_head_already_skipped`.
- Document the new CLI surface in `docs/operator-cli.md`.

## Runtime Notes

- Release promotion must restart or kick launchd after this release packet
  commit is tagged so `release:status` proves launchd is running the tagged
  head.
- The first scheduler cycle after promotion should reconcile legacy
  `processed_head_already_skipped` failed queue rows into `stale_retired` or
  `provider_deferred`, depending on the stored processed-review error.
- Provider-overload rows may still require cooldown expiry and retry before
  `release:status` turns green. Do not claim the live beta healthy while
  expired provider cooldowns or retryable provider-deferred jobs remain.
- `repoMemory.enabled=false` means this release ships the packet machinery and
  dry-run/operator surfaces first; a later release should explicitly enable
  repo memory per policy once #82 GitNexus context and #85 sticky-vs-cold eval
  gates are ready.
