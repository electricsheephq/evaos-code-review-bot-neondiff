## Purpose

Ship the first safe issue-enrichment foundation without turning on live issue
comments. Issue enrichment now has its own allowlist, throttle model, dry-run
scan command, and status visibility, separate from the PR review allowlist. The
live default remains disabled, with an empty issue allowlist and `postIssueComment`
off.

## Changes

- Add separate issue-enrichment config and validation.
- Add `issue-enrichment-scan` as a dry-run-only operator command.
- Add issue-enrichment state to `doctor`, `status`, and `runtime-inventory`.
- Keep issue enrichment independent from PR monitoring and PR review repo policy.
- Add per-repo issue-enrichment overrides, cooldowns, burst windows, lookback
  windows, and per-cycle issue/comment caps.
- Refuse invalid `issue-enrichment-scan --since` timestamps before GitHub calls.
- Avoid reading issues for repos outside the issue-enrichment allowlist or with
  per-repo issue enrichment disabled.
- Size scan pagination from the larger of per-cycle and burst thresholds, bounded
  by the operator cap.
- Include `nextEligibleAt` on deferred issue-enrichment scan rows.
- Preserve default live behavior: no issue scan repos, no issue comments, no PR
  posting policy changes, and no target-repo writes.

## Runtime Notes

- Issue enrichment is intentionally not active in live runtime. The feature now
  has a separate safety gate so PR allowlist expansion cannot accidentally enable
  issue enrichment for high-volume repos or milestone bursts.
- The GitHub App still lacks issue-read/comment permissions for live issue
  enrichment. App-token probes against bot-repo issues returned `403 Resource not
  accessible by integration`; this remains a setup blocker before any live issue
  enrichment rollout.
- Active PR monitoring currently includes 20 repos. The App read check passes for
  all 20, including `Martian-Engineering/lossless-claw`; existing open PR heads
  in that repo are activation-baselined rather than mass-reviewed.
- This release does not enable native ZCode skills, MCP, memory, tools,
  auto-merge, approvals, target-repo mutation, or live issue comments.
- This release does not increase `reviewConcurrency.maxActiveRuns`; live remains
  one-at-a-time.
